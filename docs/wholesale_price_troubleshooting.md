# 도매가(wholesale_price) 자동화 트러블슈팅 문서

> **목적**: 매일 자동 업데이트 파이프라인 구축 중, 마스터 데이터셋의 `wholesale_price`(수박 도매가, 원/kg)를 API로 재현하지 못하는 문제를 정리하고 팀 의사결정을 받기 위함.
> **작성일**: 2026-06-23
> **최종 업데이트**: 2026-06-23 — **✅ 해결됨**

## ★ 해결 결과 (최종)

도매가 출처를 찾았고 마스터와 **최근 데이터 오차 0.0**으로 재현 성공.

- **소스**: 가락 공공데이터 `dataid=data36` (단위별 도매가격). 전용 키(id=9239) 필요.
- **규칙**: `수박(일반)` + 등급 `상` 행만 → 각 행을 원/kg 환산(`1 kg`은 평균가 그대로, `N kg개`는 평균가÷N) → **단순 평균**.
- **검증**: 2026-06 구간 12일 전부 오차 0.0 완전 일치.
- **과거 드리프트 = CPI 물가보정으로 규명!** 마스터 도매가는 **실질가격**: `raw × (기준CPI / 그달CPI)`, 기준CPI=119.92(최신). CPI 보정 시 과거 오차 -279 → **±17로 수렴**(검증 완료).
  | 날짜 | 보정 전 | CPI 보정 후 |
  |------|:---:|:---:|
  | 2022-07-15 | -278.8 | -13.7 |
  | 2023-07-31 | -226.7 | +17.4 |
  | 2025-06-23 | -69.1 | -0.8 |
- **파이프라인 적용 방식 (전체 재환산 — 팀 결정)**: `wholesale_price = raw × (기준CPI / 그달CPI)`를 **매 실행마다 전 구간 재계산**.
  - **기준CPI = 수집된 CPI 중 '최신 발표월' 값 자동** (매달 갱신). CPI 월단위 발표라 그 달 내 고정, 새 달 CPI 발표 시 자동으로 새 기준. (지금은 5월 119.92)
  - **새 CPI 발표 시 전 구간이 일관된 실질가격으로 재환산** → 타겟(`wholesale_price` + 파생 `y_t`, `price_lag/ma`)이 월 단위로 바뀜 → **월 1회 모델 재학습 전제**.
  - **raw 원본 보관**: `data/price_raw.csv` sidecar(나만의 단일 진실)에 nominal raw 누적. sidecar 없으면 기존 마스터에서 1회 역산. 매 실행 raw×base/cpi로 전체 재계산 → 누적오차 없음.
  - config `cpi_base=0.0`(자동) / 양수(수동 고정).
- 구현: `collect_price.py`(raw 수집) + `merge_all_features.py`(전체 CPI 재환산 + sidecar). 합성테스트(base 변경 시 재환산) + 실데이터(2364일) 검증 완료. 5종 파이프라인 end-to-end 동작 확인.

## 3. 문제: 도매가(wholesale_price)를 API로 재현 불가

### 3-1. KAMIS로는 불가능 (검증 완료)

- KAMIS는 수박을 **"1개" 단위 가격(원/개)**로만 제공. `unit` 필드가 명시적으로 "1개".
- `p_convert_kg_yn=Y`(1kg 환산) 옵션은 **정상 작동하지만** 수박엔 적용 안 됨:
  - 감자(20kg 단위): 31,000원 → 1,550원 (÷20) ✅ 환산됨
  - 수박(1개 단위): 18,800원 → 18,800원 ❌ 변화 없음 (KAMIS가 수박 무게를 모름)
- KAMIS 원/개를 마스터 원/kg으로 나눈 "환산무게"가 **4.3~12.4kg로 매일 출렁** + 상관계수 0.67 → 단순 변환 불가, 애초에 다른 시세.
- **KAMIS에는 무게(kg)·물량 필드가 아예 없음.**

### 3-2. 팀원이 실제로 쓴 방법 (스크린샷으로 확인)

KAMIS 가격정보 화면에는 수박이 **포장단위별**로 나옴:

| 일자 | 품목 | 단위 | 등급 | 평균가 |
|------|------|------|:---:|------:|
| 2026.06.22 | 수박(일반) | 6키로개 | 상 | 14,247 |
| 2026.06.22 | 수박(일반) | 7키로개 | 상 | 17,496 |
| 2026.06.22 | 수박(일반) | 8키로개 | 상 | 20,212 |
| 2026.06.22 | 수박(일반) | 9키로개 | 상 | 20,650 |
| 2026.06.22 | 수박(일반) | 1키로 | 상 | 2,395 |
| 2026.06.22 | 수박(일반) | 10키로상자 | 상 | 21,314 |

- 팀원 방법: **"N키로개" 행만 골라 `평균가 ÷ N = 원/kg`**, `1키로`·`10키로상자`는 **수기로 제외**.
- 이건 **사람이 판단해 고른 정제 작업**이라 API로 동일 재현이 어려움.
- ❓ **이 포장단위별 화면이 KAMIS OpenAPI로도 나오는지 미확인** (우리가 호출한 KAMIS API는 "수박(1개)" 단일 행만 반환).

### 3-3. 가락 경매결과(data12)로 시도 → 가격 불일치

`data12`는 거래 건별로 `PPRICE`(경락가) + `UUN`(거래단량, "8kg") + `QTY`(수량) + `PUMJONG`(품종) + `DDD`(등급) 제공. → `원/kg = PPRICE ÷ UUN` 계산 가능.

**여러 집계 방식 시도 결과 (마스터와 대조):**

| 방식 | 2023-07-31 | 2024-08-15 | 2026-06-19 | 비고 |
|------|:---:|:---:|:---:|------|
| 마스터 (정답) | 3,374.5 | 2,236.2 | 2,344.7 | — |
| 전체 수박 가중평균 | 2,523 | 2,623 | 1,933 | 오차 큼 |
| 특(1등) 가중평균 | 2,830 | 2,744 | 2,062 | 오차 ~544 |
| 일반수박+특1등 | 2,739 | 2,535 | 1,992 | 오차 300~680 |
| KAMIS원개 ÷ 8kg | (RMSE 723) | — | — | 날마다 100~2,500 출렁 |

→ **어떤 품종/등급/평균방식 조합으로도 마스터를 재현하지 못함.** (반입량은 data12로도 근접하나, 가격은 계속 어긋남)

### 3-3-1. data12 ≠ KAMIS 게시가 (결정적)

`data12`를 단위(UUN)별로 묶어 KAMIS 스크린샷과 직접 대조한 결과 **값이 다름**:

| 단위·등급 | KAMIS 게시(스크린샷) | data12(가락경매) |
|------|:---:|:---:|
| 8키로개 / 상 | 20,212 | 15,664 |

→ 마스터(KAMIS 게시가 기반)는 **data12로 재현 불가**. 둘은 출처가 다른 별개 데이터.
→ **재학습을 피하려면(=과거와 동일 기준 유지) KAMIS "단위별 게시가"를 API로 받아야 함.** 그 데이터는: KAMIS API ❌(수박1개 통합값만) / data12 ❌(값 다름) / 서울농수산식품공사 "주요 품목 가격"(data.go.kr 15004517) ❓ 미확인(새 키 필요).

### 3-4. 추가 발견

- **법인마다 품종 표기가 다름**: 같은 일반수박을 서울청과는 `수박`, 농협/중앙은 `수박(일반)`, `수박(일반꼭지절단)`, `수박(일반)(꼭지절단)` 등으로 표기 → 단순 키워드 필터로 일관 추출 어려움.
- **무게(kg) 정보의 위치**: 반입량(4번 data22)엔 총 무게만 있고 개당 무게 없음. **개당 무게(`UUN`)는 경매결과(5번 data12)에만 존재.** → 도매가 원/kg은 5번이 유일한 후보.

## 5. 의사결정 옵션

도매가는 **예측 타겟**이라, 과거값(팀원 방식)과 미래값(API 방식)의 **기준이 반드시 일치**해야 함. 안 그러면 경계에서 값이 점프하여 모델 정확도에 직접 타격(현재 추정 오차 ~20%).

| 옵션 | 내용 | 트레이드오프 |
|------|------|------|
| **A. 과거 전체 재계산** | 도매가 전체를 하나의 API 방식으로 다시 산출 → 시리즈 전체 일관 | 타겟값 변경 → **모델 재학습 필요**. data12 과거 데이터 활용 가능 |
| **B. 팀원 방식 근접 재현** | "N키로개÷N, 박스·1키로 제외" 규칙을 API로 최대한 모사 | 약간의 offset 잔존 가능 |
| **C. 팀원 정제 코드 확보** | 기존 코드를 그대로 자동화 | 코드/노트북 필요. 가장 정확 |

**ML 관점 권장: A** (학습·서빙 타겟 정의 일치 원칙). 단, 팀 재학습 합의 필요.

## 부록: 기술 메모

- 가락 경매결과 API: `http://www.garak.co.kr/homepage/publicdata/dataJsonOpen.do`
  - 파라미터: `id`, `passwd`, `dataid=data12`, `s_date`(YYYYMMDD), `s_bubin`(법인코드), `s_pummok=수박`, `portal.templet=false`
  - 법인코드: 서울청과 11000101 / 농협(공) 11000102 / 중앙청과 11000103 / 동부팜청과 11000104 / 한국청과 11000105 / 대아청과 11000106
  - `UUN`은 `"8kg"` 형태 문자열 → 숫자 파싱 필요. 한글 파라미터는 euc-kr 인코딩.
- KAMIS: `unit="1개"`, 무게/물량 미제공. `dailyPriceByCategoryList` / `periodWholesaleProductList` 동일.
- 반입량(4번): 가락 `data22`, 수박 `PUM_CD=22100`의 `SUM_TOT`(톤) = 마스터 `trade_volume`(정수 반올림). 4개 날짜 정확 일치 검증.
